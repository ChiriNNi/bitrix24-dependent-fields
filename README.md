# Зависимые поля Bitrix24

Локальное приложение Bitrix24 для заполнения связанных данных объекта прямо в карточке сделки.

Приложение встроено не как отдельная вкладка, а как пользовательское CRM-поле приложения `UF_CRM_DEP_WIDGET` внутри карточки сделки. Менеджер выбирает клиента, город или адрес в любом порядке, а приложение сужает остальные варианты и сохраняет согласованный набор значений в реальные поля сделки.

## Текущий продакшен

- Репозиторий: `ChiriNNi/bitrix24-dependent-fields`
- Деплой: `https://bitrix24-dependent-fields.vercel.app/`
- URL приложения: `https://bitrix24-dependent-fields.vercel.app/api/app`
- URL установки: `https://bitrix24-dependent-fields.vercel.app/api/install`
- Поддерживаемые legacy routes: `/`, `/index.html`, `/app`, `/app.html`, `/install`, `/install.html`
- Основная тестовая сделка: `721747`
- Воронка, куда установщик добавляет поле: `CATEGORY_ID = 75`

## Что хранится в сделке

Приложение не хранит собственную базу. Оно записывает значения в существующие поля CRM-сделки:

| Назначение | Поле сделки |
| --- | --- |
| Адрес объекта инфо | `UF_CRM_1743501476` |
| Наименование ИП инфо | `UF_CRM_1742459776` |
| Ответственное лицо ИП инфо | `UF_CRM_1743669674` |
| Город, список | `UF_CRM_1707153439` |
| Город, текст | `UF_CRM_1747301550750` |
| Клиент | стандартное `COMPANY_ID` |

## Источники данных

Главный источник связей - список/инфоблок адресов.

| Сущность | ID |
| --- | --- |
| Адрес объекта | `IBLOCK_ID = 115` |
| Наименование ИП | `IBLOCK_ID = 109` |
| Ответственное лицо ИП | `IBLOCK_ID = 117` |

Поля адресного инфоблока:

| Назначение | Свойство |
| --- | --- |
| Чистый адрес | `PROPERTY_839` |
| Ответственное лицо | `PROPERTY_905` |
| Наименование ИП | `PROPERTY_907` |
| Компания | `PROPERTY_951` |
| Компания, fallback | `PROPERTY_1243` |
| Город | `PROPERTY_959` |

Адрес является центральной сущностью. Именно из него приложение получает клиента, город, ИП и ответственное лицо.

## Логика зависимых полей

Менеджер может начать с любого поля.

Если выбран клиент:

- адреса фильтруются по `PROPERTY_951 = COMPANY_ID`;
- список городов сужается до городов, где у клиента есть объекты;
- адресный поиск работает только внутри адресов этого клиента.

Если выбран город:

- адреса фильтруются по `PROPERTY_959`;
- клиентский список пополняется компаниями, найденными в адресах этого города;
- строгое ограничение клиентов только городом можно добавить отдельно, сейчас логика сохраняет найденные ранее варианты, чтобы не терять контекст поиска.

Если выбран адрес:

- клиент выставляется из `PROPERTY_951` или fallback `PROPERTY_1243`;
- город выставляется из `PROPERTY_959`;
- ИП подтягивается из `PROPERTY_907`;
- ответственное лицо подтягивается из `PROPERTY_905`.

При сохранении приложение вызывает `crm.deal.update` и записывает:

- `COMPANY_ID`;
- адрес;
- ИП;
- ответственное лицо;
- город текстом;
- город списком, если найдено совпадение по названию.

## Ограничения по нагрузке

Сейчас в Bitrix около `567` компаний.

Приложение не грузит все компании при пустой форме:

- `MAX_COMPANY_ROWS = 50`;
- пустой клиентский поиск берет только первую страницу компаний;
- адресная выборка ограничена `MAX_ADDRESS_ROWS = 300`;
- названия компаний дозагружаются только для уникальных `companyId`, найденных в текущих адресах.

Самая тяжелая операция - получение адресов. Но она нужна в любом случае, потому что адрес является основным источником связей.

## Установка в Bitrix24

В локальном приложении Bitrix24:

```text
Тип приложения: Серверное
Путь обработчика: https://bitrix24-dependent-fields.vercel.app/api/app
Путь для первоначальной установки: https://bitrix24-dependent-fields.vercel.app/api/install
Права: CRM, Списки, Встраивание приложений
```

Установщик делает следующее:

1. Регистрирует пользовательский тип поля `depfields` через `userfieldtype.add`.
2. Создает поле сделки `UF_CRM_DEP_WIDGET`.
3. Добавляет это поле в секцию `dependent_fields_app` карточки сделки для категории `75`.

Если тип поля уже существует, установщик вызывает `userfieldtype.update`.

## Высота iframe и белый хвост

Это главный курьез Bitrix в этой задаче.

Белая пустая область под приложением часто не является CSS-отступом внутри приложения. В DevTools она видна как внешний контейнер Bitrix:

```text
ui-entity-editor-content-block
```

Причина: высота пользовательского REST-поля задается параметром `OPTIONS.height` при регистрации типа поля через `userfieldtype.add/update`.

Сейчас высота в коде:

```js
const FIELD_HEIGHT = 360;
OPTIONS: { height: 360 }
```

Но если приложение уже было установлено с высотой `620`, Bitrix может продолжать держать старую высоту до переустановки/обновления типа поля.

Что важно:

- входящий вебхук не может выполнить `userfieldtype.update`, Bitrix возвращает `403`;
- обновить тип поля может само локальное приложение через app-auth;
- после деплоя иногда нужно открыть приложение/переустановить его и обновить карточку сделки;
- `BX24.resizeWindow` помогает iframe, но не всегда пересчитывает внешний блок карточки Bitrix.

Практический порядок, если белый хвост остался:

1. Дождаться redeploy Vercel.
2. Открыть локальное приложение в Bitrix.
3. Нажать `Переустановить`.
4. Обновить карточку сделки.
5. Если не помогло, удалить и заново добавить поле `UF_CRM_DEP_WIDGET` в вид карточки.

## Почему есть скрытые select

В UI клиент и адрес выглядят как lookup-поля: одно поле ввода и выпадающий список вариантов.

Внутри оставлены скрытые `select`:

- `#client-field`;
- `#address-field`.

Они нужны как стабильное хранилище выбранного ID. Это позволяет не ломать старую логику фильтрации и сохранения, пока визуально поле выглядит как родной Bitrix lookup.

## Почему адрес - главный

Клиент, город, ИП и ответственное лицо не являются независимыми справочниками. Их согласованная комбинация живет в адресном списке.

Поэтому правильная модель:

```text
Адрес -> клиент + город + ИП + ответственное лицо
```

Если разрешить менеджеру выбрать произвольный клиент, произвольный город и произвольное ИП без адреса, можно получить несуществующую комбинацию. Приложение старается сужать варианты так, чтобы сохранить связность данных.

## Создание сделки

В момент создания сделки у карточки еще нет настоящего `ID`, поэтому прямой вызов `crm.deal.update` невозможен. Bitrix передает в пользовательское поле приложения режим создания, но `ENTITY_VALUE_ID` может быть пустым или равным `0`.

Для этого используется двухшаговая схема:

1. Менеджер выбирает клиента, город и адрес в поле `Данные объекта`.
2. Кнопка `Сохранить` внутри приложения записывает выбранный набор как JSON-черновик в значение поля `UF_CRM_DEP_WIDGET` через `BX24.placement.call('setValue', value)`.
3. Менеджер нажимает основную кнопку `Сохранить` в карточке Bitrix.
4. После появления настоящего ID сделки приложение читает черновик из `PLACEMENT_OPTIONS.VALUE`, применяет данные в реальные поля через `crm.deal.update` и пытается очистить техническое значение поля приложения.

Это не создает фальшивый ID сделки и не требует отдельного внешнего хранилища. Черновик живет внутри создаваемой карточки Bitrix и сохраняется вместе с ней.

## Что происходит при пустой форме

При пустых фильтрах:

- грузится первая пачка компаний, максимум 50;
- грузится первая пачка адресов, максимум 300;
- город показывает общий список из CRM и городов найденных адресов.

Это сделано, чтобы менеджер мог начать с любого поля.

## REST и авторизация

В Bitrix приложение работает через `BX24.callMethod`, то есть через авторизацию установленного локального приложения и активный контекст Bitrix.

На Vercel приложение не должно хранить входящий вебхук.

Для локальной разработки используется proxy `server.js` и переменная окружения:

```powershell
$env:BITRIX_WEBHOOK_URL = "https://your-domain.bitrix24.kz/rest/user/webhook/"
$env:PORT = "4176"
node server.js
```

Локальный URL для теста:

```text
http://localhost:4176/templates/index.html?dealId=721747
```

Входящий вебхук нужен только для разработки и диагностики. В репозиторий его не добавлять.

## Vercel routes и ошибка 405

Bitrix часто открывает приложение POST-запросом. Если отдать статический `index.html`, Vercel может вернуть `405`.

Поэтому все основные входы маршрутизируются на serverless handlers:

- `api/app.js`;
- `api/install.js`.

`api/app.js`:

- читает POST/GET параметры Bitrix;
- извлекает `PLACEMENT_OPTIONS`;
- пытается определить `dealId`;
- inject-ит `window.BITRIX_REQUEST_DATA`;
- inject-ит `window.BITRIX_PLACEMENT_OPTIONS`;
- возвращает HTML с корректными frame headers.

## Как определяется сделка

Приложение пытается найти ID сделки в таком порядке:

1. `BX24.placement.info().options.ID`;
2. `ENTITY_VALUE_ID`;
3. `ENTITY_DATA.entityId`;
4. `window.BITRIX_PLACEMENT_OPTIONS`;
5. query-параметр `dealId`;
6. URL referer вида `/crm/deal/details/{id}/`.

Это сделано потому, что Bitrix в разных местах и режимах передает placement options немного по-разному.

## Известные тонкости

- `lists.element.get` не всегда корректно ищет по `%NAME`, поэтому поиск адреса делается по уже загруженной выборке.
- Фильтр по `PROPERTY_959` может вести себя нестабильно, поэтому есть fallback: загрузить более широкую выборку и отфильтровать на фронте.
- Некоторые адреса не имеют ИП/ответственного. В этом случае поля честно остаются пустыми.
- Если выбран город, клиентский dropdown может содержать ранее найденные компании. Это сделано, чтобы не ломать сценарий поиска; строгую фильтрацию клиентов по городу можно добавить отдельно.
- В Bitrix вид карточки бывает персональным. Если один пользователь видит поле, другой может не видеть, пока не добавит его в свой вид или пока конфигурация не применена в нужном scope.
- Название секции Bitrix и название нашего приложения могут дублироваться визуально. Сейчас шапка внутри iframe убрана, а заголовок секции показывает сам Bitrix.
- Если Vercel обновился, а Bitrix показывает старое, почти всегда виноват кеш iframe/карточки. Помогают hard refresh, переоткрытие сделки или переустановка локального приложения.

## Что можно улучшить дальше

- Сделать строгую фильтрацию клиентов по выбранному городу.
- Добавить индикатор загрузки внутри lookup-полей.
- Добавить отдельное сообщение, если адрес выбран, но у него нет ИП/ответственного.
- Добавить режим “создать новый адрес”, если менеджер ничего не нашел.
- Вынести ID полей и инфоблоков в отдельный конфиг.
- Добавить автоматическую проверку реального Bitrix placement после деплоя.
